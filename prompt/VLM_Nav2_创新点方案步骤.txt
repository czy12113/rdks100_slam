RDK小车创新点方案步骤：本地轻量化VLM + Nav2动态避障 + 前端实时回传 + 断网/恢复对比
生成日期：2026-07-01

一、创新点定位

核心创新点：
在RDK S100小车上部署本地轻量化VLM，使小车在公网不可用、千问API超时的情况下，仍能依靠本地视觉理解、YOLO/深度测距、Nav2代价地图完成动态行人障碍物识别、重规划绕行、近距离停车告警；恢复网络后可切回云端VLM增强对话，形成“离线自主安全决策”和“在线增强语义交互”的前后对比。

比赛展示关键词：
1. 公网断开不影响自主导航安全。
2. 本地VLM只做语义理解和告警解释，实时避障由YOLO/Depth/LiDAR/Nav2闭环完成，控制链路稳定。
3. 网页刷新后自动重连，视频、地图、检测框、VLM描述、告警和延迟/FPS指标继续实时回传。
4. 可用终端日志和curl超时证明云端不可达，但小车仍完成识别、停下、重规划、绕行。
5. 清空防火墙规则或插回网线后切回qwen_vl，展示云端增强对话效果。

三、总体架构

数据链路：
D435i RGB/Depth + Livox/scan_filtered
  -> d435i_detection：person检测、bbox、distance_m、标注画面
  -> dynamic_person_obstacle_node：把行人检测转成Nav2可消费的动态障碍点云/激光点
  -> Nav2 local_costmap/global_costmap：障碍膨胀、重规划、绕行
  -> vlm_scene internvl_local：本地VLM对关键帧做语义确认和自然语言描述
  -> safety_decision_node：近距离/横穿风险停车告警，必要时发布 /cmd_vel_estop
  -> backend WebSocket：视频、地图、路径、VLM、告警、性能指标实时推前端
  -> frontend：刷新后自动重连，同屏展示小车画面、地图、重规划状态、告警和帧率/延迟

云端/本地切换：
1. 离线比赛演示：provider:=internvl_local，本地VLM，不请求公网。
2. 断网证明：curl 千问API地址超时，但 /vlm/scene_description 仍输出本地结果。
3. 恢复网络：provider:=qwen_vl 或 openai_vision，展示云端增强对话。

四、实施步骤

阶段1：补齐本地轻量化VLM provider

目标：
把 ros2_ws/src/vlm_scene/vlm_scene/providers/internvl_local.py 从占位实现改成可运行本地推理。

推荐模型路线：
1. 首选轻量模型：InternVL2-2B / MiniCPM-V-2.x / Qwen2-VL-2B-Instruct 这类2B级别视觉语言模型。
2. 推理形态按设备性能选择：
   - 优先：RDK S100支持的BPU/HBM部署链路，模型预处理和视觉encoder尽量放硬件加速。
   - 兜底：ONNXRuntime / int8 / int4 CPU推理，限制输入分辨率和触发频率。
3. 不要让VLM跑控制闭环。VLM延迟通常高于避障控制周期，安全决策必须以YOLO/Depth/LiDAR/Nav2为实时主链，VLM只做语义确认、解释和告警文本。

需要实现：
1. internvl_local.py：
   - healthcheck 检查 VLM_LOCAL_MODEL_PATH、VLM_LOCAL_TOKENIZER、推理后端是否存在。
   - _load_local() 懒加载模型，避免节点启动直接失败。
   - _infer(req) 接收 req.frame_bgr 和 req.detections，输出中文描述。
   - 返回 VLMResponse(description, provider="internvl_local", model, elapsed_ms)。
2. vlm_params.yaml：
   - provider: "internvl_local"
   - max_image_side: 448 或 512
   - cooldown_sec: 1.0~3.0
   - min_detections: 1
   - max_detections: 5
3. 输出格式建议保持JSON或结构化文本：
   {
     "scene": "前方约1.2米有行人...",
     "risk": "stop" | "reroute" | "clear",
     "reason": "行人处于小车前方安全距离内",
     "elapsed_ms": 650
   }

ros2 launch vlm_scene vlm.launch.py provider:=internvl_local
ros2 topic echo /vlm/status --once

阶段2：动态行人障碍物接入Nav2

目标：
放置动态行人后，小车可以识别、减速/停下、更新代价地图、触发Nav2重规划并绕行。

推荐新增ROS2节点：
包位置建议：ros2_ws/src/czybot_navigation2/scripts/dynamic_person_obstacle_node.py

节点职责：
1. 订阅 /detection/results。
2. 只处理 class_name == "person" 且 confidence >= 0.55 的目标。
3. 使用 bbox中心点 + distance_m 估算行人在 base_link 或 camera frame 下的位置。
4. 发布 /dynamic_person_points，类型建议 PointCloud2。
5. 发布 /vlm/safety_event 或 /navigation/safety_event，类型 std_msgs/String(JSON)，记录：
   - event: person_detected / reroute / stop / clear
   - distance_m
   - confidence
   - action
   - timestamp
6. 如果 distance_m <= 0.8m 或 bbox位于画面中间危险区域，发布 /cmd_vel_estop 并写日志：
   [SAFETY] person too close, estop, distance=0.62m

Nav2配置修改：
在 nav2_params.yaml 的 local_costmap 和 global_costmap obstacle_layer 中增加 observation source。

示例：
observation_sources: scan dynamic_person
dynamic_person:
  topic: /dynamic_person_points
  max_obstacle_height: 2.0
  clearing: True
  marking: True
  data_type: "PointCloud2"
  obstacle_max_range: 3.0
  obstacle_min_range: 0.0
  raytrace_max_range: 3.5
  raytrace_min_range: 0.0

演示逻辑：
1. 小车导航到目标点。
2. 人从前方或侧前方进入路径。
3. detection_node 输出 person + distance。
4. dynamic_person_obstacle_node 发布动态障碍点。
5. local_costmap 出现动态障碍，MPPI减速并绕开。
6. 如果距离过近，safety node触发 /cmd_vel_estop，小车停下并前端告警。
7. 人离开后，costmap clearing 清除动态点，Nav2继续重规划到达目标。

验证命令：
ros2 topic echo /detection/results --once
ros2 topic echo /dynamic_person_points --once
ros2 topic echo /vlm/safety_event
ros2 topic echo /cmd_vel
ros2 topic echo /plan

阶段3：本地VLM参与决策解释和告警

目标：
证明“本地轻量化VLM+Nav2完成，不再请求云端”。

实现方式：
1. detection_node 和 dynamic_person_obstacle_node 做实时风险判定。
2. vlm_node 使用 internvl_local 对关键帧进行语义确认：
   - 行人是否在通道正前方。
   - 行人是否正在横穿。
   - 是否建议停车等待或绕行。
3. VLM结果发布 /vlm/scene_description。
4. safety_event 中附带 vlm_summary 或最近一次VLM描述。
5. 前端展示“本地VLM：internvl_local / elapsed_ms=xxx / risk=stop/reroute”。

建议触发策略：
1. person第一次出现：立即触发VLM。
2. person距离变化超过0.5m：触发VLM。
3. person进入1.0m危险区：强制触发VLM并停车。
4. 心跳推理间隔：10~20s，避免本地模型占满CPU/BPU。

关键日志：
[VLM] provider=internvl_local model=xxx ready=true
[VLM] infer_ok elapsed_ms=xxx trigger=person_near
[SAFETY] action=reroute reason=person_on_path distance=1.35
[SAFETY] action=stop reason=person_too_close distance=0.62
[NAV2] replan path changed old_len=xx new_len=xx

七、指标建议

必须展示的指标：
1. 网页刷新重连时间：目标 < 5s。
2. RGB回传帧率：目标 10~15fps。
3. 检测帧率/延迟：显示 infer_ms，目标现场稳定即可。
4. 本地VLM延迟：目标 0.5~3s，取决于模型和硬件。
5. 动态障碍响应：
   - person出现到前端检测显示：< 1s。
   - person进入危险区到停车：< 0.5s，实际以安全为准。
   - person离开到恢复导航：2~5s，取决于costmap clearing。
6. 云端不可用证明：curl -m 5 超时。
7. 云端恢复证明：curl成功，provider切换qwen_vl。

八、风险和兜底

1. 本地VLM太慢：
   - 降低 max_image_side 到448。
   - cooldown_sec 调大到3~5s。
   - VLM只在person出现/进入危险区时触发。
   - 安全动作仍由YOLO/Depth/Nav2执行，不等待VLM。

2. BPU模型转换来不及：
   - 使用ONNXRuntime/CPU int8或mock结构化本地provider先跑通闭环。
   - 答辩时强调provider插件化，可替换本地推理后端，比赛现场展示离线闭环行为。

3. 动态障碍不进costmap：
   - 先用 /scan_filtered 的真实激光动态障碍完成Nav2绕行。
   - dynamic_person_obstacle_node 同步发布告警和前端红点，作为语义增强。
   - 再逐步把 /dynamic_person_points 接入costmap。

4. 前端刷新后缺少VLM状态：
   - 修 DEFAULT_TOPICS 或 AUTO_SUBSCRIBED_TOPICS。
   - 页面 mounted 时调用 /api/vlm/status 和 /api/vlm/history 兜底。

5. 近距离停车不够快：
   - 不要等Nav2规划失败。
   - safety node 根据 distance_m 直接发布 /cmd_vel_estop。
   - 停车阈值建议 0.8m，恢复阈值建议 1.2m，避免抖动。

九、最终交付清单

代码交付：
1. vlm_scene/providers/internvl_local.py：本地轻量VLM真实推理。
2. czybot_navigation2/scripts/dynamic_person_obstacle_node.py：行人动态障碍和安全事件。
3. nav2_params.yaml：costmap增加 dynamic_person observation source。
4. ros2_bridge.py / data_pusher.py / websocket_manager.py：safety_event、vlm_status、navigation状态推送。
5. VideoView.vue / NavigationView.vue：本地/云端模式、FPS/延迟、动态障碍、重规划状态展示。

演示交付：
1. 离线模式视频：curl千问超时，小车仍识别行人、重规划、停车告警。
2. 恢复模式视频：清空防火墙/插回网线，切回qwen_vl，云端增强对话。
3. 终端日志截图：provider、elapsed_ms、infer_ms、safety_event、/cmd_vel_estop。
4. 前端截图：视频、地图、VLM描述、FPS、延迟、告警同屏。

一句话答辩总结：
本项目不是把大模型简单接到网页，而是把本地轻量化VLM嵌入ROS2导航安全链路：公网不可用时，小车仍可依靠本地视觉语义、目标检测、深度测距和Nav2完成动态行人识别、重规划绕行与停车告警；公网恢复后再切回云端VLM增强自然语言交互，实现离线自主安全和在线智能交互的双模式对比。

10.2 关键决策与偏差说明
-----------------------
1) 节点名统一：vlm_scene_node（不是 vlm_node）。
   backend/call_vlm_ask 原来写成 /vlm_node/set_parameters，会导致 provider 切换
   静默失败；已修复。

2) Nav2 与 safety node 的职责边界：
   实施上把"停车 / 急停"完全交给 Nav2 costmap + local planner，
   dynamic_person_obstacle_node 只负责发布 PointCloud2 + JSON 事件；
   避免多头下发 /cmd_vel 冲突。
   如后续需要"抢占式硬急停"，可以在 node 里加发 /cmd_vel_estop 但需谨慎。

3) 本地 VLM 加载失败的兜底：
   internvl_local provider 一旦模型加载失败自动切规则兜底（backend="rule_based"），
   前端徽标会展示"本地规则兜底"（黄色），确保演示不会翻车。

4) 前端 composable 单例：
   useSafetyEvent / useVlmProvider 都是模块级单例，
   AUTO_SUBSCRIBED_TOPICS 覆盖后连 vlm_status / safety_event 都不需要页面显式 subscribe，
   刷新页面立即恢复。

5) demo_offline.sh 用 iptables REJECT 而不是 DROP：
   REJECT 会让 curl 立刻收到 "connection refused"，
   截图 / 日志更明确；DROP 只会静默 timeout，演示效果差。

10.4 已知限制 & 后续工作
------------------------
- 本地 VLM 推理在纯 CPU 上耗时较长（>3s），建议启用 NPU / GPU；
  规则兜底提供保底响应，保证 UI 不阻塞。
- dynamic_person_obstacle_node 使用相机内参默认值，
  若使用非 D435i 相机需要通过参数覆盖 fx/fy/cx/cy。
- SafetyEventOverlay 与 FireAlertOverlay 目前只做视觉分层（z-index）避免遮挡，
  尚未做"火警优先级抢占安全事件"的强一致合并；如遇同时触发可以扩展。
