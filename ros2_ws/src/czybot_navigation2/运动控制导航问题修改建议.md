# 运动控制与 Nav2 导航问题修改建议

## 现象判断

当前日志中的 `controller_server: Failed to make progress` 不是单纯的规划失败。结合 `/cmd_vel` 已经出现 `linear.x ~= 0.35` 的非零速度，但小车仍然不动，首要怀疑链路是：

`Nav2 已经算出速度` -> `/cmd_vel` 话题可见 -> `stm32_bridge 是否真正收到` -> `串口是否持续写入 STM32` -> `STM32 是否驱动电机` -> `/odom` 是否随车体运动更新。

只看 `/cmd_vel` 不能证明底盘执行了命令。Nav2 的 progress checker 只看机器人位姿是否变化；如果 `/odom` 或 `odom -> base_link` 没有变化，即使 `/cmd_vel` 一直有值，也会触发 `Failed to make progress`。

## 必须优先确认的问题

1. 当前运行的包可能不是源码中的最新配置。

   本工作区中 `src/czybot_navigation2/launch/livox_navigation.launch.py` 已存在，但 `install/czybot_navigation2/share/czybot_navigation2/launch/` 里没有该文件；`install/.../config/nav2_params.yaml` 也是 2026-05-31 的旧版本。`ros2 launch czybot_navigation2 ...` 默认从 install/share 读取文件，所以必须先确认实际机器 `sunrise@ubuntu` 上加载的是哪个 overlay。

   建议现场执行：

   ```bash
   ros2 pkg prefix czybot_navigation2
   ros2 pkg prefix czybot_slam
   ros2 launch czybot_navigation2 livox_navigation.launch.py --show-args
   ```

   如果 prefix 不是当前 `~/rdks100_slam/ros2_ws/install/czybot_navigation2`，说明你改的源码和实际运行的不是同一份。

2. `/cmd_vel` 发布者/订阅者必须唯一且明确。

   现场执行：

   ```bash
   ros2 topic info /cmd_vel -v
   ros2 node info /stm32_bridge
   ```

   正常期望：

   - `/cmd_vel` 至少有 `/stm32_bridge` 订阅者。
   - 最终向 `/cmd_vel` 发布速度的节点最好只有一个。
   - 如果同时看到 `controller_server`、`velocity_smoother`、`collision_monitor` 都在向 `/cmd_vel` 或相关话题发布，需要重新整理速度链路。

3. 必须验证 `/odom` 是否随命令变化。

   单独绕开 Nav2 测：

   ```bash
   ros2 topic echo /odom
   ros2 topic pub -r 10 /cmd_vel geometry_msgs/msg/Twist "{linear: {x: 0.25}, angular: {z: 0.0}}"
   ```

   如果 `/cmd_vel` 有值但小车不动、`/odom.twist.twist.linear.x` 也不变化，问题在底盘桥接、串口、STM32 协议、电机使能或下位机保护，不应继续优先调 Nav2。

## 已发现的配置风险

1. install 目录是旧配置。

   旧 `install/.../nav2_params.yaml` 中仍有这些风险项：

   - `enable_stamped_cmd_vel: true`
   - `collision_monitor.cmd_vel_out_topic: /diff_drive_controller/cmd_vel`
   - costmap 仍订阅 `/scan`，不是 `/scan_filtered`
   - planner 仍是旧频率和旧插件配置

   这些都会导致 `stm32_bridge` 收不到最终速度，或 Nav2 使用了和当前源码不一致的感知/控制链。

2. 源码中的速度链路注释和参数仍不完全一致。

   `src/czybot_navigation2/config/nav2_params.yaml` 里注释说 `velocity_smoother` 目前是旁路，但 `collision_monitor` 又配置为：

   ```yaml
   cmd_vel_in_topic: "cmd_vel_smoothed"
   cmd_vel_out_topic: "/cmd_vel"
   ```

   如果 controller 仍直接发 `/cmd_vel`，同时 smoother/collision_monitor 也参与发布，实际 `/cmd_vel` 可能出现多发布者或旁路链路，调试会很混乱。

3. `stm32_bridge` 的串口发送策略可能不适合 STM32 看门狗。

   源码中 `_serial_write_thread()` 在非零速度时会循环写串口，但如果上层命令中断超过 `cmd_timeout=0.3s` 会主动发零速。需要确认 Nav2 输出频率、bridge 写频率、STM32 `CMD_TIMEOUT_MS` 三者一致。若现场 `/cmd_vel` 有短暂停顿，STM32 会立刻停车，Nav2 随后判定无进展。

## 建议修改顺序

1. 先重新构建并确认运行的是最新包。

   ```bash
   cd ~/rdks100_slam/ros2_ws
   colcon build --packages-select czybot_navigation2 czybot_slam --symlink-install
   source install/setup.bash
   ros2 pkg prefix czybot_navigation2
   ```

2. 把速度链路改成单一路径，建议使用：

   `controller_server /cmd_vel_nav` -> `velocity_smoother /cmd_vel_smoothed` -> `collision_monitor /cmd_vel` -> `stm32_bridge`

   对应修改方向：

   - 在 `livox_navigation.launch.py` 给 `controller_server` remap：`cmd_vel -> cmd_vel_nav`
   - 给 `velocity_smoother` remap：`cmd_vel -> cmd_vel_nav`，`cmd_vel_smoothed -> cmd_vel_smoothed`
   - 保持 `collision_monitor.cmd_vel_in_topic: "cmd_vel_smoothed"`
   - 保持 `collision_monitor.cmd_vel_out_topic: "/cmd_vel"`
   - `stm32_bridge` 只订阅最终 `/cmd_vel`

   修改后 `/cmd_vel` 应该只有 collision_monitor 一个发布者，`stm32_bridge` 一个订阅者。

3. 如果只是验证底盘，先临时禁用 `velocity_smoother` 和 `collision_monitor`。

   最小链路：

   `controller_server /cmd_vel` -> `stm32_bridge`

   这样可以快速判断“不动”是不是 Nav2 后处理链路造成的。底盘能动后，再恢复 smoother 和 collision_monitor。

4. 给 `stm32_bridge` 增加可观测性。

   建议增加一个 debug topic，例如 `/stm32_bridge/tx_cmd`，发布实际写入串口的 `linear_mm/s` 和 `angular_mrad/s`；或者至少每 1 秒 info 打印一次最近发送值。这样可以区分：

   - `/cmd_vel` 有值但 bridge 没收到
   - bridge 收到但被死区/急停/超时清零
   - bridge 写了串口但 STM32 没执行

5. 现场验证顺序固定为四步。

   ```bash
   ros2 topic info /cmd_vel -v
   ros2 topic echo /cmd_vel
   ros2 topic echo /odom
   ros2 run tf2_ros tf2_echo odom base_link
   ```

   判断标准：

   - `/cmd_vel` 非零、`/odom` 不变：底盘执行链路问题。
   - `/cmd_vel` 为零或断续：Nav2 控制链或 collision_monitor 问题。
   - `/odom` 变但 TF 不变：`stm32_bridge` TF 发布问题。
   - `/odom` 和 TF 都变，但 Nav2 仍报 progress failed：AMCL/map/local costmap 或 progress checker 参数问题。

## 结论

当前优先级不是继续调 MPPI 的角速度细节，而是先修正"实际运行文件"和"速度命令链路"。你的 `/cmd_vel` 已经能看到 0.35 m/s，但车不动，说明根因更靠近 `stm32_bridge -> 串口 -> STM32 -> 电机/odom`。只有确认手动 `/cmd_vel` 能让车动、`/odom` 同步变化后，再继续调 Nav2 的 MPPI、progress checker 和 collision_monitor。

## v4 补充：小车冲过目标点 + 来回摆动问题

### 现象

`2D Goal Pose` 派任务后小车能动，但速度过快、冲过目标点继续向前；之后会返回，最终在目标点附近来回摆动。直线测试就出现，且摆动幅度明显大于 `xy_goal_tolerance`。

### 根因

这是典型的 **MPPI 输出与底层执行模型不匹配** 导致的超调，三处叠加：

1. `stm32_bridge.min_motion_linear = 0.18` 把所有 0 < |v| < 0.18 m/s 的命令一律抬到 0.18 m/s。
   MPPI 在终点前的减速命令（0.05~0.18 m/s）全部被强制抬回 0.18 m/s，车永远以最低 0.18 m/s 冲过目标。
2. `MPPI.vx_max = 0.35` 配合上面的 0.18 启动门槛，可用调速带宽仅 [0.18, 0.35]，
   减速可分辨率不足 0.17 m/s，配合 `xy_goal_tolerance=0.50` 的宽容差，必然冲过头。
3. `PreferForwardCritic.threshold_to_consider = 0.8 / cost_weight = 4.0` 让 MPPI 在距目标 80 cm 内仍强烈倾向"向前"，
   压制了 GoalCritic 的减速作用。

### v4 修复（已落地）

修改了三个文件，必须 `colcon build --packages-select czybot_navigation2 --symlink-install` 后重启才能生效。

#### 1. [`config/nav2_params.yaml`](config/nav2_params.yaml:1)

| 参数 | 旧值 | 新值 | 作用 |
|---|---|---|---|
| `MPPI.vx_max` | 0.35 | **0.22** | 巡航速度从 ≈1.3km/h 降到 ≈0.8km/h |
| `MPPI.vx_min` | -0.18 | **-0.12** | 倒车更慢，避免后退超调 |
| `MPPI.wz_max` | 0.80 | **0.60** | r_min = 0.22/0.60 ≈ 0.37 m，与 AckermannConstraints 对齐 |
| `MPPI.ax_min` | -1.0 | **-1.5** | 减速度加大，能在 1 个 model_dt 内刹住 |
| `MPPI.vx_std` | 0.10 | **0.06** | 候选轨迹更聚焦低速减速带 |
| `MPPI.prune_distance` | 1.7 | **1.0** | 前瞻收紧，避免远端路径压制近端目标 |
| `GoalCritic.cost_weight` | 12.0 | **20.0** | 终点吸引力翻倍，主导终点段控制 |
| `GoalCritic.threshold_to_consider` | 2.0 | **2.5** | 距目标 2.5m 就开始减速 |
| `PreferForwardCritic.cost_weight` | 4.0 | **2.0** | 终点附近不再硬推前进 |
| `PreferForwardCritic.threshold_to_consider` | 0.8 | **0.4** | 仅在最后 40cm 才考虑前向偏好 |
| `general_goal_checker.xy_goal_tolerance` | 0.50 | **0.20** | 容差从半米收紧到 20cm |
| `general_goal_checker.yaw_goal_tolerance` | 0.80 | **0.50** | 仍允许 ≈28° 偏差 |
| `velocity_smoother.max_velocity` | [0.35, 0, 0.80] | **[0.22, 0, 0.60]** | 与 MPPI 同步 |
| `velocity_smoother.max_decel` | [-1.0, 0, -1.2] | **[-1.5, 0, -1.2]** | 与 MPPI 同步 |

#### 2. [`launch/livox_navigation.launch.py`](launch/livox_navigation.launch.py:1)

新增 launch 参数 `stm32_min_motion_linear`，默认 **0.10**，并在 `stm32_bridge_node` 的 `parameters` 中显式注入：

```python
'min_motion_linear': 0.10,   # 旧值 0.18，让 MPPI 减速段真正生效
'linear_deadzone': 0.005,
'angular_deadzone': 0.010,
```

可调建议：
- 如果实测车在 0.10 m/s 还能稳定起步，可继续降到 **0.08**（减速带宽更大，更精准）。
- 如果 0.10 m/s 命令电机原地不动，再升到 **0.12**。
- 不要恢复到 0.18，否则超调问题必然复现。

### 还想更精准（可选）

如果 v4 修复后小车仍有 10~20cm 的过冲（可接受，但还想再优化）：

- 或在 `MPPI.critics` 列表加入 `"VelocityDeadbandCritic"`，对接近目标的高速候选轨迹直接惩罚。

不建议把 `vx_max` 降到 0.18 以下：会逼近 `min_motion_linear` 的启动门槛，调速空间被吃干，反而起步抖动。

## v5 补充：到点"停下→倒车→摆动→突然前冲"问题

### 现象

v4 修复后小车不再大幅冲过目标，但在目标点附近出现新行为：
1. xy 差不多到目标位置 → 停下
2. 立刻倒退一段
3. 来回摆动几次
4. 突然往前冲

### 根因（关键）

阿克曼小车 **不能原地旋转**，但 v4 的 `general_goal_checker.yaw_goal_tolerance=0.50` 还要求到点时车头朝向接近目标朝向。流程是：

1. xy 进入 0.20 m 容差 → goal_checker 看 yaw 偏差超 28° → **判定还没到达**
2. MPPI 为纠正 yaw 必须开弧线 → `vx_min=-0.12` 允许它瞬间倒车 → "倒退"
3. 倒一段 yaw 仍不够 → 再前进 → "来回摆动"
4. 最后 `progress_checker` 15s 无进展 → BT 触发清 costmap + 重规划 → 新路径让小车冲向目标 → "突然前冲"

### v5 修复（核心思路）

**阿克曼车的 `2D Goal Pose` 应该只判位置，不判朝向**。

修改了三处：

#### 1. [`config/nav2_params.yaml`](config/nav2_params.yaml:1) — goal_checker 禁用 yaw 检查

```yaml
general_goal_checker:
  xy_goal_tolerance: 0.20
  yaw_goal_tolerance: 3.14    # ★ 等价禁用，阿克曼不强求朝向
```

如果业务确实需要朝向对准（如对接充电桩），用 `docking_server` 或独立对接 action，**不要**用通用 Nav2 goal_checker。

#### 2. [`config/nav2_params.yaml`](config/nav2_params.yaml:1) — 禁止 MPPI 倒车

```yaml
FollowPath:
  vx_max: 0.15        # 0.22 → 0.15，慢点更准
  vx_min: 0.0         # ★ 关键：从 -0.12 改成 0.0，强制 forward-only
  wz_max: 0.50
  ax_max: 0.5
  ax_min: -1.2        # 减速度从 -1.5 调缓，避免急刹反向
  vx_std: 0.04        # 候选轨迹聚焦低速带
  wz_std: 0.15

  GoalAngleCritic:
    enabled: false    # ★ 关闭，不再让 MPPI 在终点纠正朝向

  PreferForwardCritic:
    cost_weight: 5.0          # 恢复中等权重
    threshold_to_consider: 0.5
```

`vx_min: 0.0` 是 v5 最关键的一行：阿克曼车在 Nav2 中允许倒车会带来连锁副作用——
- 到点小偏差时 MPPI 立刻发倒车命令"修正"，正是用户看到的"停下→倒退"。
- 倒车 + 舵机延迟 + bridge 启动门槛 → 容易摆动。

强制 forward-only 后，小偏差由 BT 重规划绕一圈解决，**永远不原地倒车**。如果业务必须支持后退（如脱困），通过独立 BackUp action 控制。

#### 3. [`config/nav2_params.yaml`](config/nav2_params.yaml:1) — progress_checker 放宽

```yaml
progress_checker:
  required_movement_radius: 0.05   # 0.10 → 0.05
  movement_time_allowance: 30.0    # 15.0 → 30.0
```

避免低速 + 终点小振荡被误判为"无进展"，触发不必要的 BT recovery → 突然前冲。

#### 4. [`config/nav2_params.yaml`](config/nav2_params.yaml:1) — velocity_smoother 同步

```yaml
velocity_smoother:
  max_velocity: [0.15, 0.0, 0.50]
  min_velocity: [0.0, 0.0, -0.50]    # 线速度下限设为 0
  max_accel:    [0.5, 0.0, 1.0]
  max_decel:    [-1.2, 0.0, -1.0]
```

#### 5. [`launch/livox_navigation.launch.py`](launch/livox_navigation.launch.py:1) — bridge 启动门槛

```python
DeclareLaunchArgument(
    'stm32_min_motion_linear',
    default_value='0.08',     # 0.10 → 0.08
)
```

减速带宽从 [0.10, 0.22] → [0.08, 0.15]，减速分辨率更细。


## v6 补充：小车撞向障碍物 + 撞墙后停一会又前冲

### 现象

走直线测试：

1. 建图固定中心起步，地图正中心前方一段距离有一个障碍物。
2. 导航时把车移到"中心偏后"位置，在 RViz 点 **2D Pose Estimate** 校准。
3. 设置 **2D Goal Pose** 在距障碍物前方有一段距离的位置（直线在前）。
4. 小车启动后**没有在目标点停下，直接撞向障碍物**；撞后停了一会，又往前冲。

### 根因（三层叠加）

| # | 缺陷 | 现象映射 |
|---|---|---|
| 1 | **AMCL 强制初始位姿在 (0,0,0)**：v5 设置了 `set_initial_pose: true` + `initial_pose=(0,0,0)`。即使用户在 RViz 重新校准，AMCL 启动瞬间已经把位姿"锁"在原点，导致 local costmap 用错误位姿投影激光观测，**障碍物在 costmap 中的位置和真实位置错位** | 看到的路径"避开了"costmap 中的障碍物，但车撞上真实障碍物 |
| 2 | **collision_monitor 被完全旁路**：v5 注释中说"最小验证链路"，cmd_vel_out_topic 写到无人订阅的 `/cmd_vel_collision_monitor`，等价**禁用了避障兜底**。MPPI 的 CostCritic 是软约束（cost_weight 仅 3.81，远低于 PathAlignCritic=14），定位偏一点就撞 | "直接撞向障碍物" |
| 3 | **inflation_radius 太小 + progress_checker 过宽**：inflation_radius 仅 0.18m（车体 0.15 + 缓冲 3cm），物理刹停距离都不够；`movement_time_allowance=30s` 让小车撞墙堵转后**长达 30s 才触发 BT recovery**，期间 controller 持续发 cmd_vel，STM32 看门狗一恢复就再次前冲 | "停一会又往前冲" |

### v6 修复（核心思路）

**三处协同**：定位严格化 + 真正的物理避障兜底 + 更激进的 MPPI 避障代价 + 更敏感的进度判定。

#### 1. [`config/nav2_params.yaml`](config/nav2_params.yaml:84) — AMCL 关闭强制原点定位

```yaml
amcl:
  ros__parameters:
    set_initial_pose: false   # ★ 从 true 改回 false
    # 不再写 initial_pose 字段
```

**强制要求**用户在 RViz 点 **2D Pose Estimate** 校准后再发任务。这是最稳的做法，避免 AMCL 用错位的初始姿态污染 costmap。

#### 2. [`config/nav2_params.yaml`](config/nav2_params.yaml:594) — collision_monitor 真正串入控制链路

```yaml
collision_monitor:
  cmd_vel_in_topic: "cmd_vel"           # 旧值: cmd_vel_smoothed (旁路)
  cmd_vel_out_topic: "cmd_vel_safe"     # 旧值: /cmd_vel_collision_monitor (旁路)
  FootprintApproach:
    time_before_collision: 1.5          # 旧值: 1.2
    min_points: 4                       # 旧值: 6（更敏感）
    visualize: True                     # 旧值: False（在 RViz 可视化便于调试）
```

链路从"旁路"变成"实串"：

```
controller_server → /cmd_vel
  → collision_monitor → /cmd_vel_safe
    → stm32_bridge → STM32
```

collision_monitor 的 `approach` 模式会根据前方激光距离把线速度按比例缩放，**1.5s 物理避障兜底**——即使 MPPI 没刹住也能停下。

#### 3. [`launch/livox_navigation.launch.py`](launch/livox_navigation.launch.py:271) — stm32_bridge remap

```python
stm32_bridge_node = Node(
    ...
    remappings=[('cmd_vel', 'cmd_vel_safe')],   # ★ v6 新增
    ...
)
```

这样不需要修改 nav2_bringup 内部，只通过 stm32_bridge 的 remap 把 collision_monitor 串入。

#### 4. [`config/nav2_params.yaml`](config/nav2_params.yaml:307) — CostCritic 权重大幅提升

```yaml
CostCritic:
  cost_weight: 18.0          # 旧值 3.81，超过 PathAlignCritic=14（避障优先于贴路径）
  consider_footprint: false  # ★ v6.1 修正：保持 false（见下方踩坑 #4）
  critical_cost: 220.0       # 旧值 300（更早判定近障碍为关键，提前减速）
```

这是 MPPI 层的避障加强：让 MPPI 在距障碍 30~40cm 时就主动减速绕行。

#### 5. [`config/nav2_params.yaml`](config/nav2_params.yaml:384) — inflation_radius 大幅加大

```yaml
local_costmap.inflation_layer:
  inflation_radius: 0.40     # 旧值 0.18
  cost_scaling_factor: 2.0   # 旧值 3.0（衰减更缓避免狭窄通道）

global_costmap.inflation_layer:
  inflation_radius: 0.40     # 同步
  cost_scaling_factor: 2.0
```

物理刹停距离计算：MPPI 10Hz + 巡航 0.15m/s + 减速度 1.2 m/s² + 控制延迟 100ms = 最坏 17cm 内能刹住。**新 inflation 多留 25cm 余量**，足以应对定位误差和雷达延迟。

#### 6. [`config/nav2_params.yaml`](config/nav2_params.yaml:193) — progress_checker 收紧

```yaml
progress_checker:
  required_movement_radius: 0.05
  movement_time_allowance: 8.0   # 旧值 30.0
```

撞墙堵转后 **8s 即触发 BT recovery → 清 costmap 重规划**，而不是无脑再次冲向同一障碍。

### 现象

## v6.3 修复（MPPI 评分链失衡导致小车径直撞向障碍物）

### 现象

v6.2 链路通了，`/cmd_vel` 拓扑正确、`rx > 0/s`、`tx > 0/s`，2D Goal Pose 派任务后小车正常出发，但**依然径直冲向障碍物**：
- RViz 里 local_costmap 能看到障碍物前一团蓝紫色膨胀代价区
- 全局规划画出的红线是直线穿过障碍物
- MPPI 候选轨迹也是直线
- 障碍物 ≥ 15 cm（不是 LaserScan 切片漏检）

### 根因（v6.2 的失误）

阅读 `nav2_mppi_controller/src/critics/cost_critic.cpp` 源码后定位出三层失误：

#### 真凶 1：[`CostCritic.near_goal_distance: 1.0`](config/nav2_params.yaml:353) 静默禁用避障

MPPI `CostCritic::score()` 末尾：

```cpp
if (near_goal_distance_ > 0.0 &&
    utils::withinPositionDistance(data.path, last_pose, near_goal_distance_))
    return;  // 整条轨迹的避障代价不计入评分！
```

含义：候选轨迹末端到全局路径末端距离 < 1.0 m 时，CostCritic 直接 `return`，**整条轨迹的避障代价不计入评分**。

用户测试场景：障碍物前 1~2 m 处打 Goal Pose → 几乎所有候选轨迹末端都落在这个 1.0 m 窗口 → CostCritic 形同虚设 → MPPI 只看 GoalCritic + PathAlign + PreferForward → 三者合力直指目标 → 撞。

#### 真凶 2：[`CostCritic.critical_cost: 220.0`](config/nav2_params.yaml:350) 报警太迟

InflationLayer 代价随距离指数衰减（`cost_scaling_factor=2.0`、`inscribed_radius=0.15`）：

| 距障碍中心 | 代价 |
|---|---|
| 0.15 m（inscribed） | 253（致命） |
| 0.20 m | ~228 |
| 0.25 m | ~206 |
| 0.30 m | ~186 |
| 0.40 m | ~153 |
| 0.60 m | ~83 |

`critical_cost=220` 对应车体中心距障碍 ~0.21 m（**车边缘只剩 6 cm**），到这里才"判为关键"已经来不及刹车。

#### 真凶 3：[`CostCritic.trajectory_point_step: 2`](config/nav2_params.yaml:354) 候选轨迹漏检 50%

MPPI 每条候选轨迹每 step=2 个点采样一次代价。轨迹 30 个点只查 15 个，中段穿障碍的细窄轨迹有 50% 概率漏检。

#### 帮凶 4：GoalCritic / PathAlignCritic / PreferForwardCritic 权重压过 CostCritic

| Critic | v6.2 权重 | 含义 |
|---|---|---|
| GoalCritic | 20.0（2.5 m 内激活） | 奔向目标 |

## v6.4 修复（AMCL 漂移 + 雷达高度 TF）

### 现象

v6.3 上车后避障已经生效，小车在障碍前 30 cm 左右停下。但触发 1 次 `clear_costmap` recovery 后，RViz 里看到障碍物轮廓**整体平移 20~30 cm**（白色"原位置"→ 蓝色"新位置"）。

### 根因

不是"到点重新校准"，而是 **AMCL 全程持续漂移** + clear_costmap 把漂移一次性"显形"：

```
t=0  : 2D Pose Estimate 校准，AMCL 收敛
t=N  : 阿克曼小车转弯打滑 + 编码器误差，里程计累积漂移
       AMCL 默认 update_min_d=0.25 才更新一次 → 1.67s 内漂移不被纠正
       v6.3 默认 max_particles=800 + alpha1~5=0.2 → 粒子云不够、噪声模型太信任里程计
t=N+1: 避障停车 → BT 触发 clear_costmap → 用当前(已漂移的) AMCL 位姿重建 costmap
       → 障碍物画在新位置 → 看起来"图跳了"
```

漂移幅度判断标准：

| 漂移量 | 评价 | 影响 |
|---|---|---|
| < 5 cm | 优秀 | 几乎察觉不到 |
| 5~15 cm | 可接受 | 不影响到点和避障 |
| 15~30 cm | **偏大**（v6.3 实测处于此区间） | 重复任务可能"绕道"或贴墙 |
| > 30 cm | 失效 | 必须重新校准 |

### v6.4 修复

#### 1. [`config/nav2_params.yaml`](config/nav2_params.yaml:39) — 阿克曼专用噪声模型
